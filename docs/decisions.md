## [2026.03.26] 어드민 정책 수립

**결정**: 관리자 계정을 위한 어드민 패널과 Sidebar Admin 탭 구현

**이유**:
- 서비스 운영을 위해 사용자/피드백/카테고리 등을 관리할 필요
- 어드민 계정은 이미 있지만 UI에서 접근 불가능

**정책**:
1. **어드민 식별**: `profiles.is_admin = true`
2. **UI 위치**: Sidebar 네비게이션 (is_admin=true일 때만 표시)
3. **페이지**: /admin/users, /admin/categories, /admin/feedback, /admin/docs
4. **접근 제어**: 로그인 필수 + is_admin 체크 (middleware + admin/layout)

**대안**:
1. 대시보드 배너 — 항상 눈에 띄지만 배너 공간 점유
2. 프로필 드롭다운 — 숨겨져 있어서 모르기 쉬움
3. Sidebar Admin 탭 ✅ — 자연스러움, 네비 구조와 통일, 모바일에서 자동 숨김

**트레이드오프**:
- Sidebar와 profiles 쿼리 필요 (minor — 성능 무시할 수 있음)
- 모바일에서는 노출 안 됨 (대부분 데스크톱에서 사용하므로 OK)

**구현**:
- [src/components/common/Sidebar.jsx](src/components/common/Sidebar.jsx): is_admin 쿼리 + Admin 탭 조건부 렌더링
- Shield 아이콘 + "관리자" 레이블

**트레이드오프**:
- `position: fixed` 사용으로 z-index 관리 필요
- 모바일 safe-area 고려 필수

**관련 커밋**: `64635d0` (feat: FeedbackButton 리스타일링)

**이유**:

- 소수 사용자 피드백 전에 미리 검증하고 싶음

1. 사용자 피드백 기다리기 — 너무 늦을 수 있음
2. 자동화된 A/B 테스트 — 아직 사용자 부족
3. 휴리스틱 평가 ✅ — 저비용, 빠른 발견

**발견된 이슈 (5개 수정)**:
- 게스트 NULL 세션 노출 (dashboard, history)
- 게스트 /study/new 접근 불가 (로그인 유도 모달 추가)
- 랜딩 CTA 게스트 지원 미부족
- /history 탭 혼란 (완료 세션만 표시로 변경)
- 세션 카드 상태 뱃지 개선

**트레이드오프**:
- 개발 일정 단축 vs 품질 검증 시간
- 결론: 지인 피드백 전 반드시 해야 함

**관련 커밋**: `7291b6f~7484260`

## [2026.03.17] 학습 플로우 재설계

**결정**: 첫학습 / 이어서(복습 or 바로) 진입 분기 방식으로 변경

**이유**:
- 기존 플로우가 신규 사용자와 재방문자 구분 안 됨
- 중단 세션 복습 여부를 명시적으로 선택하게 하고 싶음
- "진행률"을 명확히 하려면 문항 수 기반 진도율 필요

**변경 사항**:
1. 첫학습 → 스킵, 이어서 → 복습/바로 선택
2. 선택형(4지선다) UI — 선택 강조 + 300ms 딜레이 후 자동 전송
3. 진도율 표시 — "N문항 학습 중"
4. Score 분리 — 학습 중 숨김, 완료 시만 평균 저장
5. sessionStorage 키 분리 — roadmap별 독립

**대안**:
1. 기존 플로우 유지 — 혼란스러운 UX 지속
2. 강제 첫학습 → 진행 막음
3. 현재 방식 ✅ — 명확한 선택지

**트레이드오프**:
- 로직 복잡도 증가 (첫/이어 분기)
- 기존 users의 expectation 변화
- 재방문율 향상 예상

**관련 커밋**: `6cc042a~7291b6f`

## [이전] 교안 기능 (v0.2 시도 후 Revert)

**결정**: 챕터/스텝 분할 방식 교안 기능을 구현했으나 revert

**이유**:
- 정확한 기록을 남기지 않아서 지금은 뭐가 문제였는지 불명확
- 의도대로 작동 안 됨
- "작은 단위부터 만들고 정리"라는 원칙 위배 (scope creep)

**작업한 내용** (전부 revert):
- `/api/curriculum/generate` — 챕터 목록 생성 API
- `CurriculumPanel.jsx` — 목차 사이드 패널
- 챕터 인디케이터, 프로그레스 바, "다음 챕터로" 버튼

**배운 점**:
- 기록 없음 = 나중에 같은 실수 반복
- 목적이 흔들리면 (빌드 vs 보여주기) scope 늘어남
- MVP → 피드백 → v0.3의 순서가 맞음

**회고**: "지금 내가 만드는 건가, 보여주려는 건가?" 자문 필요

**교안은 v0.3으로 미룸**: 스텝 분할 방식 재설계 예정

## [2026.02] AI 모델 선택: Haiku vs Sonnet

**결정**: 개발/테스트는 claude-haiku-4-5, 프로덕션은 claude-sonnet-4-6

**이유**:
- 개발 속도: Haiku 비용 1/10 (빠른 반복)
- 품질: Sonnet이 필요하면 배포 전 교체

**모델별 특성**:
| 모델 | 응답시간 | 품질 | 비용 | 용도 |
|------|---------|------|------|------|
| Haiku | 빠름 | 중상 | 낮음 | 개발/테스트 |
| Sonnet | 중간 | 높음 | 중간 | 프로덕션 |

**대안**:
1. Sonnet만 사용 — 비용 10배
2. 항상 Haiku — 품질 부족 가능성
3. 현재 방식 ✅ — 비용 vs 품질 균형

**트레이드오프**:
- 프로덕션 배포 전 모델 교체 필수 (환경변수 수정)
- 개발/프로덕션 동작 미묘한 차이 가능성 (배포 전 재테스트)

## [2026.02] 스트리밍 방식: SSE vs WebSocket

**결정**: Server-Sent Events (SSE) 방식 채택

**이유**:
- Next.js API Route 기본 지원
- HTTP 표준 (크로스 브라우저)
- 타이핑 효과 구현 간단

**구현 방식**:
```
AI → "data: {chunk}" → DONE 이벤트 → 메타데이터 전송
```

**대안**:
1. WebSocket — 양방향 통신 불필요 (과도)
2. Long polling — 방화벽 문제 가능성
3. SSE ✅ — 간단하고 표준

**트레이드오프**:
- 재연결 로직 필요 (연결 끊김 시)
- 서버 메모리 증가 (open connection 유지)

## [2026.02] 인증: Supabase Auth 선택

**결정**: Supabase Auth (Google/GitHub/Magic Link)

**이유**:
- PostgreSQL RLS와 통합
- OAuth, Magic Link 기본 지원
- 별도 인증 서버 불필요

**지원 방식**:
- Google OAuth ✅
- GitHub OAuth ✅
- 이메일+비밀번호 ✅
- Magic Link (token_hash) ✅
- 게스트 모드 (user_id = NULL) ✅

**Magic Link 선택 사유**:
- PKCE 방식은 크로스 브라우저 문제 발생
- token_hash 방식 사용

**대안**:
1. Auth0 — 비용
2. Firebase Auth — Google 강제
3. Supabase ✅ — 통합, 저비용

**트레이드오프**:
- Supabase에 vendor lock-in
- 게스트 데이터 보안 (user_id IS NULL 누구나 조회 가능)

## [2026.02] DB: 게스트 데이터 (user_id = NULL)

**결정**: 게스트 모드 허용, user_id = NULL 데이터 저장

**이유**:
- 로그인 강제 시 이탈률 높음
- 체험 후 로그인 유도가 효과적

**대안**:
1. 로그인 필수 — 이탈률 증가
2. 익명 auth (random UUID) — 나중에 마이그레이션 복잡
3. 현재 방식 ✅ — 즉시 구현 가능

**트레이드오프**:
- RLS: user_id IS NULL 데이터는 누구나 조회 가능 (보안 이슈)
- 해결책: v0.3에서 익명 auth로 마이그레이션 예정

## [2026.02] DB: RLS 정책 설계

**결정**: categories/sessions/reviews 모두 user_id 기반 RLS

**정책**:
```sql
-- categories
user_id IS NULL OR auth.uid() = user_id

-- sessions / reviews
auth.uid() = user_id OR user_id IS NULL
```

**이유**:
- 기본 카테고리 (user_id NULL) 모두 공유
- 로그인 사용자는 본인 데이터만 접근
- 게스트는 본인 + 공유 데이터

**트레이드오프**:
- 게스트 데이터 보안 (해결책: 익명 auth)
- 카테고리 공유 불가 (v1에서)
